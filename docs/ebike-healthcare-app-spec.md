# 전기자전거 헬스케어 앱 — 제품 기획 및 결정 사항

> 대화를 통해 정리한 아이디어, 현실성 검토, 확정 결정을 기록한 문서입니다.  
> 최종 갱신: 2026-06-24

---

## 1. 제품 개요

### 1.1 한 줄 정의

GPS·심박·(선택) 수동 태그로 **「얼마나 내가 밟았는지」**와 **「얼마나 모터에 맡겼는지」**를 추정하고, 그 패턴으로 **운동량·칼로리·배터리 소모**를 함께 추적하는 안드로이드 앱.

### 1.2 해결하려는 문제

전기자전거는 페달을 밟을 수도 있고 안 밟을 수도 있어, 일반 피트니스 앱이 GPS 거리만으로 계산하는 **운동량·칼로리가 부정확**하다. 모터 의존도를 분리해 **실제 몸을 쓴 만큼**을 보여주는 것이 목표다.

### 1.3 타깃 사용자

전기자전거로 출퇴근·레저를 겸용하며, 「운동은 했는지 애매한」 사용자.

### 1.4 차별점

1. **E-bike 전용** — 자력 vs 모터 구간 분리
2. **운동 + 에너지** — 몸 쓴 만큼과 배터리 쓴 만큼을 한 화면에
3. **점점 똑똑해지는 개인화** — 라이딩·배터리 기록이 쌓일수록 추정·예측 개선

## 2. 핵심 아이디어

### 2.1 PAS vs 스로틀 구분

- **PAS** (Pedal Assist System): 페달 보조. 페달을 밟아야 모터가 도움.
- **스로틀**: 손잡이 등으로 모터만 구동. 페달 없이 주행 가능.

앱은 모터 컨트롤러에 직접 연결하지 않는 한 **PAS/스로틀을 100% 판별할 수 없다.**  
따라서 **직접 감지가 아닌 추론(inference)** 으로 **자력 기여 비율(0~100%)** 을 구간별로 추정한다.

#### 사용 신호

| 신호 | 용도 | 한계 |
|------|------|------|
| **HR** (심박, Heart Rate) | 페달 시 상승·유지 vs 스로틀 시 상대적 저유지 | 더위·긴장·카페인·언덕 긴장 |
| **GPS** (속도·가속·거리) | 주행 패턴 분석 | 강한 PAS도 비슷한 패턴 |
| **고도/경사** | 오르막에서 HR·속도 관계 | 내리막 페달링은 HR 낮음 |
| **수동 태그** | 사용자가 직접 정답 제공 | 선택 기능, 안 눌러도 동작 |

#### 구간 분류 (개념)

```
ACTIVE_PEDAL    — HR·속도·경사가 자력 주행 패턴
MOTOR_DOMINANT  — 저노력 고속, HR 대비 과도한 속도 등
COAST / IDLE    — 거의 정지 또는 관성 주행
```

### 2.2 운동량·칼로리

- **핵심 KPI: 실페달 km** — 전체 주행 중 실제 페달한 거리 환산 (자력 추정 비율 × 거리의 합).
- 부가 지표(추후): 유효 운동 시간, 추정 칼로리(MET·HR 기반, 모터 구간 가중치 감소).
- UI에는 **추정치**임을 명시. 의료·진단 목적이 아님.

### 2.3 배터리 소모·잔여 주행 거리 예측

사용자의 PAS/스로틀(자력) 사용 패턴과 배터리 기록을 학습해 **Wh/km** 소모를 추정하고, 남은 에너지로 주행 가능 거리를 예측한다.

- 초기(라이딩 5~10회 미만): 「학습 중 · 신뢰도 낮음」 표시.
- 전압만으로는 SOC가 부정확하므로 **용량(Wh) + 측정 규칙**을 함께 사용.

## 3. 확정 결정 사항

| 항목 | 결정 |
|------|------|
| **핵심 KPI** | **실페달 km** |

## 4. 배터리 입력 상세

### 4.1 최초 설정 (1회)

- 배터리 **정격 전압** (예: 36V / 48V / 52V)
- 배터리 **용량** — **Ah** 또는 **Wh** (라벨·구매 스펙, `Wh ≈ V × Ah`)
- 완충 후 **휴식 상태 전압(V)** — 충전기 분리 후 **30분 휴식** 뒤 측정 권장

### 4.2 라이딩 종료 후

- **V** (휴식 **10분** 후 측정 권장) **또는**
- 자전거 디스플레이 **SOC%** (State of Charge, 잔량 %) 중 편한 것 입력  
- 둘 다 있으면 정확도에 유리

### 4.3 이후 단계 (MVP 이후)

- BLE로 BMS/디스플레이에서 SOC% 자동 수집 (지원 모델 한정)

### 4.4 용어

| 용어 | 의미 |
|------|------|
| **V** | 전압 (볼트) |
| **Ah** | 암페어시 — 용량 단위 |
| **Wh** | 와트시 — 에너지 양 (`Wh = V × Ah`) |
| **SOC%** | 충전 상태 %, 디스플레이 잔량 |

## 5. 오프라인 정책

### 5.1 정의

- **필수:** 셀룰러·Wi‑Fi **없이** 라이딩 기록·실페달 km·종료 리포트·배터리 입력까지 동작.
- **범위:** 폰을 거치대에 두고 라이딩하는 전제. 워치 단독(폰 없이) 주행은 MVP 제외.
- Galaxy Watch 심박은 **인터넷이 아닌 블루투스**로 폰과 연동.

### 5.2 라이딩 중 반드시 오프라인 동작

- 라이딩 시작 / 일시정지 / 종료
- GPS 궤적·거리·속도·(가능하면) 고도
- 구간 분류 → **실페달 km** 누적 (HR 없어도 GPS + 수동 태그로 최소 동작)
- 화면: 실페달 km, 주행 거리, 시간
- 로컬 DB 저장 (주기적 flush, 복구 가능)
- 종료 후 배터리 V / % 입력

### 5.3 선택·제한

| 항목 | 정책 |
|------|------|
| 심박 반영 | BT 연결 시. 지연·배치 동기화 허용, 구간 단위 소급 반영 |
| 지도 | 라이딩 중 없음. **종료 후** 궤적 표시 (온라인 시 타일 로드) |
| 클라우드·계정 | 온라인 시에만 (MVP 필수 아님) |

### 5.4 HR 지연 대응

서드파티 앱에 HR이 실시간(1초 단위)으로 오지 않을 수 있음.  
라이딩 중 UI는 「현재 순간 HR」보다 **최근 구간 평균**을 전제로 하고, 종료 후 HR 지연 보정 재계산을 허용.

## 6. 수동 태그

### 6.1 개념

라이딩 중 사용자가 **지금의 주행 모드**를 직접 알려주는 버튼. 기록 구간에 **라벨(태그)** 을 붙인다.

### 6.2 버튼 (안)

```
[ 페달만 ]  [ 스로틀 ]  [ 혼합 / 모름 ]
```

- **페달만** — 스로틀 없이 페달 위주
- **스로틀** — 페달 거의 없이 스로틀 위주
- **혼합** — 둘 다 또는 애매할 때

### 6.3 동작 원칙 (확정)

- **버튼을 안 눌러도** 라이딩 및 자동 추정 계산은 **정상 동작**.
- **누르면** 해당 구간부터 그 모드로 기록되며, **실페달 km 정확도·개인 패턴 학습**에 도움.
- 모드는 라이딩 중 **언제든 변경** 가능.

### 6.4 사용 예

1. 출발 → [페달만]
2. 피곤 → [스로틀]
3. 다시 운동 → [페달만]
4. 종료 → 구간별 합산으로 실페달 km 산출

## 7. Galaxy Watch 연동

- 경로: **Galaxy Watch → Samsung Health → (Health Connect) → 앱**
- 비행기 모드 + 블루투스 ON 환경에서 동작 목표
- 라이딩 **필수 조건 아님** — HR 없으면 GPS·자동 추정·수동 태그만으로 동작, HR 있으면 정확도 상승

## 8. 지도 (B안)

| 시점 | 동작 |
|------|------|
| **라이딩 중** | 지도 미표시. 실페달 km·거리·시간 등 **숫자 중심** UI |
| **라이딩 종료 후** | GPS 궤적 **지도 표시** (기록은 오프라인, 지도 타일은 연결 시 로드) |

오프라인 라이딩 엔진과 KPI(실페달 km)에 집중하고, 지도는 **사후 확인** 용도.

## 9. MVP (Minimum Viable Product)

**최소 기능 제품** — 처음 출시할 때 꼭 필요한 기능만 포함한 1차 버전.

### 9.1 MVP 포함

- [ ] 라이딩 기록 (GPS, 오프라인, 포그라운드 서비스)
- [ ] **실페달 km** (핵심 KPI, 홈·라이딩 화면 강조)
- [ ] Galaxy Watch 심박 연동 (Health Connect / Samsung Health, BT)
- [ ] 자동 구간 추정 (HR + GPS + 고도)
- [ ] **수동 태그** (선택, 미사용 시에도 동작)
- [ ] 배터리: 정격 V + Ah/Wh 설정, 종료 후 V 또는 SOC%
- [ ] 라이딩 종료 리포트
- [ ] **종료 후** 궤적 지도 (B안)
- [ ] 로컬 DB (Room 등), 오프라인 퍼스트

### 9.2 MVP 이후

- 오프라인 지도 타일 캐시
- 배터리 잔여 주행 거리 예측 (학습 데이터 축적 후)
- BLE 케이던스 센서 / e-bike 디스플레이 연동
- 클라우드 백업·주간 리포트
- 워치 단독 라이딩 (Wear 앱)
- 개인 보정 모델 고도화

## 10. 아키텍처 원칙 (개념)

```
[라이딩 중]
  GPS 센서 ──┐
  HR (BT)  ──┼──► 로컬 세그먼트 엔진 ──► 실페달 km (실시간 갱신)
  수동태그 ──┘              │
                            ▼
                    Room DB (주기적 flush)

[네트워크 복귀 후]
  (선택) 클라우드 백업 / HR 지연 보정 재계산 / 종료 후 지도 타일
```

- **Single source of truth = 로컬 DB**
- 네트워크 실패해도 라이딩은 실패하지 않음
- 모든 수치는 **추정치**로 표현, 신뢰도 표시 검토

## 17. 개발 시 유의사항 / 해야 할 것 / 하지 말아야 할 것 (Android 중심)

### 유의할 점

- **추정치 전제**: PAS/스로틀은 100% 판별이 아니라 “자력 기여 비율 추정”. 단정 표현 금지.
- **HR 지연/누락 기본값**: Health Connect/워치 연동은 불안정할 수 있다. HR 없이도 동작해야 한다.
- **오프라인 퍼스트**: 라이딩 중 네트워크는 신뢰하지 않는다. 로컬에서 기록/저장/리포트 완결.
- **배터리 입력 UX 우선**: 전압은 흔들린다. “휴식 후 V/SOC% 입력” 규칙을 UX로 강제/안내.
- **MVI 규율**: State/Event/Effect(또는 Side-effect) 경계를 흔들면 급격히 복잡해진다.

### 해야 할 것

- **권한/백그라운드 선행 해결**: 위치(정확/백그라운드), 포그라운드 서비스, 배터리 최적화 예외 안내를 초기에 설계.
- **MVP는 1회 사용자 여정 완성**: 시작→기록→종료→리포트(종료 후 지도)까지 끊김 없이.
- **Room 데이터 모델 먼저**: 라이딩/세그먼트/태그/배터리의 최소 필드부터 확정.
- **핵심 계산은 테스트로 보호**: 실페달 km, 분류 로직은 단위 테스트 대상.

### 하지 말아야 할 것

- **“정확한 PAS/스로틀 구분”을 약속하지 않기**
- **HR 없으면 앱이 동작하지 않게 만들지 않기**
- **라이딩 중 온라인 지도 의존하지 않기** (B안 유지: 종료 후 지도)

## 18. 권한/포그라운드 서비스 설계 (minSdk 30 기준)

이 앱의 라이딩 기록은 **백그라운드에서의 지속 위치 업데이트**가 핵심이므로, Android 정책과 UX를 먼저 고정한다.

### 18.1 Manifest 권한 (초안)

- **필수(라이딩 기록)**
  - `android.permission.ACCESS_FINE_LOCATION` (정확한 위치)
  - `android.permission.ACCESS_COARSE_LOCATION` (대략 위치 — 플랫폼 요구상 함께 선언하는 경우가 많음)
  - `android.permission.FOREGROUND_SERVICE`
  - `android.permission.FOREGROUND_SERVICE_LOCATION` (Android 10+에서 위치 FGS)

- **권장(알림)**
  - `android.permission.POST_NOTIFICATIONS` (Android 13+ 런타임 권한)

- **선택(워치/헬스 데이터)**
  - Health Connect 권한(런타임): 심박(HR) 읽기 등. (Manifest 권한이 아니라 Health Connect API에서 요청)

> 참고: minSdk=30이라도 targetSdk가 최신이므로, Android 13+ 단말에서는 알림 권한 흐름이 필요하다.

### 18.2 런타임 권한 요청 순서 (UX 플로우)

라이딩 화면 진입/시작 시점에 한 번에 다 받으려 하지 말고, “왜 필요한지”를 먼저 설명한다.

1. **온보딩/설명 화면**
   - “라이딩 기록을 위해 위치 권한이 필요합니다”
   - “라이딩 중에도 기록을 유지하기 위해 ‘라이딩 기록 중’ 알림이 표시됩니다”

2. **정확한 위치 권한 요청**
   - `ACCESS_FINE_LOCATION` (필수)

3. **알림 권한 요청(해당 단말에서만)**
   - Android 13+에서 `POST_NOTIFICATIONS`
   - 알림이 거부되더라도 “라이딩 기록 자체”는 가능하게 하되, 포그라운드 서비스 UX는 재설계 필요(사용자에게 알림 허용 유도)

4. **(필요 시) 배터리 최적화 예외 안내**
   - 기기/상황에 따라 GPS가 끊기면 “배터리 최적화 해제” 안내 화면으로 유도
   - 권한이 아니라 시스템 설정이므로, 사용자가 선택하도록 안내

5. **Health Connect 권한(선택)**
   - “HR이 있으면 실페달 km 추정 정확도가 올라갑니다(선택)”
   - 거부/실패 시에도 라이딩은 진행

### 18.3 포그라운드 서비스 설계 (라이딩 엔진)

- **원칙**
  - 라이딩 중에는 반드시 **포그라운드 서비스 + 지속 알림**으로 위치 업데이트를 유지한다.
  - 서비스는 “라이딩 기록” 단일 책임으로 두고, UI/추정 엔진/저장을 분리한다.

- **수명**
  - 라이딩 시작 → 서비스 시작
  - 일시정지 → 위치 업데이트 간격 최소화 또는 중단(정책 선택)
  - 종료 → 서비스 중지 + 최종 저장/리포트 생성

### 18.4 MVI 관점 분리(권장)

- **Event**: `OnStartRideClicked`, `OnPauseClicked`, `OnStopClicked`, `OnPermissionResult(...)`, `OnHealthConnectGranted(...)`
- **Effect**(사이드이펙트): `RequestLocationPermission`, `RequestNotificationPermission`, `OpenBatteryOptimizationSettings`, `StartRideForegroundService`, `StopRideForegroundService`, `RequestHealthConnectPermissions`
- **State**: 권한 상태(Granted/Denied), 서비스 상태(Running/Stopped), 라이딩 상태(Idle/Recording/Paused/Stopping)

이렇게 두면 “권한/서비스/설정 이동”이 UI 코드에 퍼지지 않고, 테스트 가능한 형태로 유지된다.

## 19. 챌린지형 피트니스 UI/UX 구체안 (Ebike Pulse 적용)

목표는 “기록 → 목표(챌린지) → 성취” 루프를 만들되, MVP에서는 리워드/커뮤니티를 최소화하고 **실페달 km** 중심으로 구성한다.

### 19.1 하단 탭(IA) 제안

- **홈**
  - 오늘 요약(실페달 km, 주행거리, 라이딩 시간)
  - 최근 라이딩 카드(마지막 1회)
  - 주간/월간 요약 진입
- **챌린지**
  - 개인 챌린지(예: 이번 주 실페달 km 목표)
  - 챌린지 달성 현황/배지(로컬)
- **라이딩**
  - 라이딩 시작/일시정지/종료(핵심 플로우)
  - 수동 태그(페달만/스로틀/혼합)
- **기록**
  - 라이딩 목록(날짜별)
  - 라이딩 상세(종료 후 지도 + 실페달 km 리포트)
- **설정**
  - 배터리(정격 V, Ah/Wh), Health Connect, 권한/배터리 최적화 안내, 단위/개인정보

> MVP에서는 “커뮤니티/리워드” 대신 **챌린지(개인 목표)**로 동기부여 구조를 먼저 구현한다.

### 19.2 홈 화면 레이아웃(카드 구성)

“시간대/요약 카드” 패턴을, e-bike 라이딩에 맞게 단순화한다.

- **Hero KPI 카드(상단 고정)**
  - 큰 숫자: **오늘 실페달 km**
  - 보조: 오늘 주행거리 / 오늘 라이딩 시간
  - 우측 작은 상태: HR 연동 상태(연결됨/미연동) + 데이터 품질(낮음/보통/높음)

- **최근 라이딩 카드**
  - 제목: “마지막 라이딩”
  - 요약: 실페달 km / 주행거리 / 시간 / (선택) 평균 HR
  - 썸네일: 종료 후 지도 스냅샷(온라인이면 타일, 오프라인이면 폴리라인만)

- **주간 카드(동기부여 영역)**
  - 이번 주 실페달 km 누적 + 목표 대비 진행률(챌린지와 연결)
  - “이번 주 목표 설정/변경” 진입

- **배터리 카드(선택)**
  - “최근 라이딩 종료 배터리 기록” 유도(입력 미완료 배지)
  - 배터리 예측은 MVP에서는 숨기거나 “학습 중”으로 표시

### 19.3 라이딩 화면(기록 화면) 레이아웃

라이딩 중에는 지도 없이 숫자 중심(B안 유지). “실시간 카운터” 감각을 가져간다.

- 상단: 상태(Recording/Paused) + 시간
- 메인: **실페달 km(큰 숫자)** / 주행거리 / 현재 속도
- 보조: (가능하면) 최근 구간 HR / 고도(또는 누적 상승고도는 나중)
- 하단: [일시정지] [종료] + 수동 태그 토글(페달만/스로틀/혼합)

### 19.4 종료 후 리포트 화면(“마무리 리포트” 패턴)

“하루가 마무리되면 통계”를 보여주는 것처럼, Ebike Pulse는 “라이딩이 끝나면 리포트”를 강하게 제공한다.

- 상단: 지도(폴리라인) + 기본 요약
- KPI: 실페달 km / 자력 비율(%) / 주행거리 / 시간
- 입력 유도: “배터리 V 또는 SOC% 기록(선택)” CTA
- 데이터 품질: HR 미연동/태그 미사용 등으로 신뢰도 낮으면 안내 문구

### 19.5 권한/연동 온보딩(안내 문구 구조)

- 위치 권한: “라이딩 기록(경로/거리)을 위해 필요”
- 알림(해당 단말): “라이딩 중 기록 유지를 위해 ‘라이딩 기록 중’ 알림이 표시”
- Health Connect(HR): “선택 기능이며, 허용 시 실페달 km 추정 정확도 향상”
