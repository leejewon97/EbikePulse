## 1. 개발 가이드 (Android 중심)

### 유의할 점

- **추정치 전제**: PAS/스로틀은 100% 판별이 아니라 “자력 기여 비율 추정”. 단정 표현 금지.
- **HR 지연/누락 기본값**: Health Connect/워치 연동은 불안정할 수 있다. HR 없이도 동작해야 한다.
- **오프라인 퍼스트**: 라이딩 중 네트워크는 신뢰하지 않는다. 로컬에서 기록/저장/리포트 완결.
- **배터리 입력 UX 우선**: 전압은 흔들린다. “휴식 후 V/SOC% 입력” 규칙을 UX로 강제/안내.
- **MVI 규율**: State/Event/Effect(또는 Side-effect) 경계를 흔들면 급격히 복잡해진다.

### 해야 할 것

- **권한/백그라운드 선행 해결**: 위치(정확/백그라운드), 포그라운드 서비스, 배터리 최적화 예외 안내를 초기에 설계.
- **MVP는 1회 사용자 여정 완성**: 시작→기록→종료→리포트(종료 후 지도)까지 끊김 없이.
- **Room 데이터 모델 먼저**: 라이딩/세그먼트/태그/배터리의 최소 필드부터 확정.
- **핵심 계산은 테스트로 보호**: 실페달 km, 분류 로직은 단위 테스트 대상.

### 하지 말아야 할 것

- **“정확한 PAS/스로틀 구분”을 약속하지 않기**
- **HR 없으면 앱이 동작하지 않게 만들지 않기**
- **라이딩 중 온라인 지도 의존하지 않기** (B안 유지: 종료 후 지도)

## 2. 권한/포그라운드 서비스 설계 (minSdk 30 기준)

이 앱의 라이딩 기록은 **백그라운드에서의 지속 위치 업데이트**가 핵심이므로, Android 정책과 UX를 먼저 고정한다.

### 2.1 Manifest 권한 (초안)

- **필수(라이딩 기록)**
  - `android.permission.ACCESS_FINE_LOCATION` (정확한 위치)
  - `android.permission.ACCESS_COARSE_LOCATION` (대략 위치 — 플랫폼 요구상 함께 선언하는 경우가 많음)
  - `android.permission.FOREGROUND_SERVICE`
  - `android.permission.FOREGROUND_SERVICE_LOCATION` (Android 10+에서 위치 FGS)

- **권장(알림)**
  - `android.permission.POST_NOTIFICATIONS` (Android 13+ 런타임 권한)

- **선택(워치/헬스 데이터)**
  - Health Connect 권한(런타임): 심박(HR) 읽기 등. (Manifest 권한이 아니라 Health Connect API에서 요청)

> 참고: minSdk=30이라도 targetSdk가 최신이므로, Android 13+ 단말에서는 알림 권한 흐름이 필요하다.

### 2.2 런타임 권한 요청 순서 (UX 플로우)

라이딩 화면 진입/시작 시점에 한 번에 다 받으려 하지 말고, “왜 필요한지”를 먼저 설명한다.

1. **온보딩/설명 화면**
   - “라이딩 기록을 위해 위치 권한이 필요합니다”
   - “라이딩 중에도 기록을 유지하기 위해 ‘라이딩 기록 중’ 알림이 표시됩니다”

2. **정확한 위치 권한 요청**
   - `ACCESS_FINE_LOCATION` (필수)

3. **알림 권한 요청(해당 단말에서만)**
   - Android 13+에서 `POST_NOTIFICATIONS`
   - 알림이 거부되더라도 “라이딩 기록 자체”는 가능하게 하되, 포그라운드 서비스 UX는 재설계 필요(사용자에게 알림 허용 유도)

4. **(필요 시) 배터리 최적화 예외 안내**
   - 기기/상황에 따라 GPS가 끊기면 “배터리 최적화 해제” 안내 화면으로 유도
   - 권한이 아니라 시스템 설정이므로, 사용자가 선택하도록 안내

5. **Health Connect 권한(선택)**
   - “HR이 있으면 실페달 km 추정 정확도가 올라갑니다(선택)”
   - 거부/실패 시에도 라이딩은 진행

### 2.3 포그라운드 서비스 설계 (라이딩 엔진)

- **원칙**
  - 라이딩 중에는 반드시 **포그라운드 서비스 + 지속 알림**으로 위치 업데이트를 유지한다.
  - 서비스는 “라이딩 기록” 단일 책임으로 두고, UI/추정 엔진/저장을 분리한다.

- **수명**
  - 라이딩 시작 → 서비스 시작
  - 일시정지 → 위치 업데이트 간격 최소화 또는 중단(정책 선택)
  - 종료 → 서비스 중지 + 최종 저장/리포트 생성

### 2.4 MVI 관점 분리(권장)

- **Event**: `OnStartRideClicked`, `OnPauseClicked`, `OnStopClicked`, `OnPermissionResult(...)`, `OnHealthConnectGranted(...)`
- **Effect**(사이드이펙트): `RequestLocationPermission`, `RequestNotificationPermission`, `OpenBatteryOptimizationSettings`, `StartRideForegroundService`, `StopRideForegroundService`, `RequestHealthConnectPermissions`
- **State**: 권한 상태(Granted/Denied), 서비스 상태(Running/Stopped), 라이딩 상태(Idle/Recording/Paused/Stopping)

이렇게 두면 “권한/서비스/설정 이동”이 UI 코드에 퍼지지 않고, 테스트 가능한 형태로 유지된다.

### 2.5 MVI 코드 조직 (Contract)

- feature당 `*Contract.kt`에 `UiState`, `Event`, `Effect` 타입을 둔다. (현재 예: `ui/ride/RideContract.kt`)
- **초기·소규모 화면**은 Contract **한 파일**로 유지한다. 타입 수가 적을 때 파일을 미리 나누지 않는다.
- 아래 **2개 이상** 해당되면 `*Event.kt`, `*Effect.kt` 등으로 **분리 검토**한다. (숫자는 가이드라인이며 절대 규칙이 아니다.)
  - Contract가 읽기 불편할 정도로 길어짐 (대략 100줄 이상)
  - `Event` 또는 `Effect` 항목이 많아짐 (대략 10개 전후)
  - 한 종류 타입만 자주 수정하거나, 같은 파일에서 merge 충돌이 반복됨
- 분리 시에도 §2.4의 **State/Event/Effect 개념 분리**는 유지한다. bootstrap 단계 번호와 무관하게 **필요할 때** 진행한다.

## 3. UI/UX 구체안

목표는 “기록 → 목표(챌린지) → 성취” 루프를 만들되, MVP에서는 리워드/커뮤니티를 최소화하고 **실페달 km** 중심으로 구성한다.

### 3.1 하단 탭(IA) 제안

- **홈**
  - 오늘 요약(실페달 km, 주행거리, 라이딩 시간)
  - 최근 라이딩 카드(마지막 1회)
  - 주간/월간 요약 진입
- **챌린지**
  - 개인 챌린지(예: 이번 주 실페달 km 목표)
  - 챌린지 달성 현황/배지(로컬)
- **라이딩**
  - 라이딩 시작/일시정지/종료(핵심 플로우)
  - 수동 태그(페달만/스로틀/혼합)
- **기록**
  - 라이딩 목록(날짜별)
  - 라이딩 상세(종료 후 지도 + 실페달 km 리포트)
- **설정**
  - 배터리(정격 V, Ah/Wh), Health Connect, 권한/배터리 최적화 안내, 단위/개인정보

> MVP에서는 “커뮤니티/리워드” 대신 **챌린지(개인 목표)**로 동기부여 구조를 먼저 구현한다.

### 3.2 홈 화면 레이아웃(카드 구성)

“시간대/요약 카드” 패턴을, e-bike 라이딩에 맞게 단순화한다.

- **Hero KPI 카드(상단 고정)**
  - 큰 숫자: **오늘 실페달 km**
  - 보조: 오늘 주행거리 / 오늘 라이딩 시간
  - 우측 작은 상태: HR 연동 상태(연결됨/미연동) + 데이터 품질(낮음/보통/높음)

- **최근 라이딩 카드**
  - 제목: “마지막 라이딩”
  - 요약: 실페달 km / 주행거리 / 시간 / (선택) 평균 HR
  - 썸네일: 종료 후 지도 스냅샷(온라인이면 타일, 오프라인이면 폴리라인만)

- **주간 카드(동기부여 영역)**
  - 이번 주 실페달 km 누적 + 목표 대비 진행률(챌린지와 연결)
  - “이번 주 목표 설정/변경” 진입

- **배터리 카드(선택)**
  - “최근 라이딩 종료 배터리 기록” 유도(입력 미완료 배지)
  - 배터리 예측은 MVP에서는 숨기거나 “학습 중”으로 표시

### 3.3 라이딩 화면(기록 화면) 레이아웃

라이딩 중에는 지도 없이 숫자 중심(B안 유지). “실시간 카운터” 감각을 가져간다.

- 상단: 상태(Recording/Paused) + 시간
- 메인: **실페달 km(큰 숫자)** / 주행거리 / 현재 속도
- 보조: (가능하면) 최근 구간 HR / 고도(또는 누적 상승고도는 나중)
- 하단: [일시정지] [종료] + 수동 태그 토글(페달만/스로틀/혼합)

### 3.4 종료 후 리포트 화면(“마무리 리포트” 패턴)

“하루가 마무리되면 통계”를 보여주는 것처럼, Ebike Pulse는 “라이딩이 끝나면 리포트”를 강하게 제공한다.

- 상단: 지도(폴리라인) + 기본 요약
- KPI: 실페달 km / 자력 비율(%) / 주행거리 / 시간
- 입력 유도: “배터리 V 또는 SOC% 기록(선택)” CTA
- 데이터 품질: HR 미연동/태그 미사용 등으로 신뢰도 낮으면 안내 문구

### 3.5 권한/연동 온보딩(안내 문구 구조)

- 위치 권한: “라이딩 기록(경로/거리)을 위해 필요”
- 알림(해당 단말): “라이딩 중 기록 유지를 위해 ‘라이딩 기록 중’ 알림이 표시”
- Health Connect(HR): “선택 기능이며, 허용 시 실페달 km 추정 정확도 향상”
